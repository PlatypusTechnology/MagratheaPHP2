# MagratheaPHP2, made for vibing with AI — the simple version

You built a solid framework. It's held up for 10 years. But it was designed for a human typing every line, often with the admin panel generating boilerplate for you. Now that Claude writes the boilerplate, some of that machinery is dead weight, and a few of the framework's habits actively work against how an AI reads and writes code.

Here's the short version of what I'd change.

## Quick suggestions (do these first, low risk)

- **Fix `Query::Clean()`.** It only escapes quotes with a backslash — that's not real SQL escaping and can be bypassed with multi-byte tricks. Replace it with `mysqli_real_escape_string` or just delete it and force everyone through prepared statements.
- **Fix `Query::BuildWhere()`.** Every `->Where(["field" => $value])` call drops the value straight into the SQL string (`` `field` = '$value' ``), no escaping at all. This is the single biggest injection risk in the framework — and it's the *recommended* pattern in your own docs.
- **Stop hand-rolling JWT header parsing** — you already depend on `firebase/php-jwt`, which is fine, but double check `getAuthorizationHeader()` doesn't have custom parsing that duplicates what the library already validates.
- **Delete the "generated Base class" comment/warning.** Since Claude — not the admin panel — writes these files now, "FILE GENERATED BY MAGRATHEA, don't hand edit" is actively misleading and will make an AI agent hesitate to touch the file or, worse, try to find the generator tool that produced it.

## Highly recommended implementations

- **Collapse Model + Control + Base + Base into one class per feature.** Right now, one table needs 4 files (`ArticleBase`, `Article`, `ArticleControlBase`, `ArticleControl`) with a lot of ritual. An AI can absolutely write and maintain a single `Article` class with static query helpers on it — the split existed so the *generator* wouldn't clobber your hand-written code. You don't need that anymore.
- **Add real query parameter binding to the fluent `Query` builder**, not just to the raw model `Insert`/`Update`/`Delete`. Today only the ORM's own low-level insert/update/delete use prepared statements; anything built through `Query::Select()->Where(...)` is vulnerable. This is the highest-value security fix in the whole codebase.
- **One config format.** Right now config is a hand-rolled `.ini`-like parser (`ConfigFile.php`) with a home-grown "environment section" system. Standard `.env` + `vlucas/phpdotenv` (or even native PHP `parse_ini_file` with real sections) is something an AI already knows cold, with zero chance of hallucinating your custom syntax.
- **Write a `MagratheaPHP2.md` "AI contract" file** (like a CLAUDE.md, shipped with the framework) that states in one page: model shape, how queries get built, where secrets live, and what NOT to hand-roll. This is cheaper than any code change and pays off every single session.

## Costly changes (bigger, but worth planning for)

- **Retire the Admin-panel code generator (`CodeCreator`, `ObjectManager`, the "objects-create/edit" admin UI) as the primary way to define models.** Keep the admin CRUD *viewer* (still useful for managing data), but stop treating the GUI wizard as the source of truth for schema. Claude reading/writing a plain PHP class is now faster and more reliable than clicking through forms.
- **Replace the custom `Database`/`Query` layer with (or wrap it around) a well-known abstraction** — Doctrine DBAL or even plain PDO with your query-builder as a thin convenience layer on top. Every additional hand-rolled abstraction is something the AI has to learn fresh instead of already knowing.
- **Normalize the Admin panel's server-rendered PHP views** (dozens of `.php` files under `src/Admin/views`) into a smaller number of components. This isn't broken, just large — an AI has to read a lot of files to understand one screen.

## I'm crazy — go big

- **Split the framework in two**: a thin "Magrathea Core" (DB access, auth, models) and a fully optional "Magrathea Admin" package. Most new apps built with AI assistance won't need the visual admin panel at all — the AI *is* the admin panel now.
- **Auto-generate an OpenAPI spec from `MagratheaApi` route definitions** instead of the custom `ApiExplorer` admin feature. Claude (and every other tool) already understands OpenAPI natively; a bespoke explorer UI doesn't help it.
- **Ditch the ORM's magic `__get`/`__set` property proxying entirely** and go to explicit typed properties with real PHP 8 constructor promotion. It's a rewrite, but it removes the single biggest source of "the AI can't tell what's real without running the code" in the framework.
